Clinical System Enhancement - Unit-Aware Capacity Tracking
Goal
Transform the Njala University Clinic reception system from a basic patient check-in and queue management application into a comprehensive unit-aware capacity tracking system that addresses:

Multi-unit clinic operations: Support for 10+ units (Emergency, Laboratory, Pharmacy, Antenatal, Post-natal, Male/Female/Children Wards, PHD, Reception)
Staff & resource constraints: Track 20 staff members across roles, bed/mattress availability, and critical equipment (oxygen, stretchers)
Capacity-aware workflows: Bed reservations, admission waitlists, equipment requests, and unit-based triage
Operational visibility: Real-time analytics for bed utilization, staff-to-patient ratios, queue metrics, and incident reporting
The system currently handles patient registration, check-in, queue management, and attendance records. This enhancement adds the critical operational layer needed for resource-constrained clinical settings.

User Review Required
IMPORTANT

Breaking Changes to AttendanceLog Model The existing AttendanceLog model will be extended with new fields (destination_unit, is_outpatient, admission_reserved, priority_level). While backward compatible (all fields nullable except defaults), this changes the check-in workflow semantics. Existing attendance records will default to is_outpatient = true and destination_unit = null (interpreted as Outpatient/Reception).

WARNING

Queue Number Logic Change Queue numbering will become unit-specific rather than global daily numbering. This affects queue board display and patient expectations. Consider communication strategy for rollout.

IMPORTANT

New Required Data Entry: Staff Roster The system will require daily/shift-based staff roster entry (StaffShift table) to enable capacity-aware wait time estimates and staff-to-patient analytics. Without this data, wait time estimates will fall back to static defaults.

Automated Tests
Currently, no automated test suite exists in the codebase. For this iteration, verification will be manual with potential for future test coverage.

Manual Verification
1. Database Schema Verification
Steps:

Run migration: cd server && npx prisma migrate dev --name add_unit_capacity_tracking
Verify tables created: Open SQL Server Management Studio, connect to SQLEXPRESS02, expand database, verify presence of:
Staff, Unit, StaffShift, Bed, BedReservation, Resource, ResourceRequest, AdmissionWaitlist, IncidentReport tables

Run seed: cd server && npx prisma db seed
Verify seed data: Query SELECT * FROM Unit (expect 10 rows), SELECT * FROM Staff (expect ~20 rows), SELECT * FROM Bed (expect ~15 rows)
Expected Result: All tables exist, seed data populated correctly.

2. Unit-Aware Check-In Flow
Scenario A: Outpatient Check-In (Default)

Steps:

Start server: cd server && npm start
Start client: cd client && npm run dev
Login as receptionist
Search for existing patient (or register new one)
Click "Check In"
In check-in modal:
Leave unit as "Outpatient" (default)
Enter visit reason
Click "Confirm Check-In"
Verify response shows queue_number, is_outpatient: true, admission_reserved: false
Navigate to Queue Board tab
Verify patient appears in queue with "Outpatient" badge
Expected Result: Patient checked in as outpatient, no bed operations triggered.

Scenario B: Emergency Check-In with Equipment Request

Steps:

Register new patient or search existing
Click "Check In"
In modal:
Select unit: "Emergency"
Check "Oxygen needed?"
Priority should auto-set to "High"
Click "Confirm Check-In"
Verify:
Patient appears in queue board with "Emergency" badge, "High" priority
Check database: SELECT * FROM ResourceRequest WHERE visit_id = <visit_id> - expect row with resource_id for oxygen, fulfilled = 1 if oxygen available
If oxygen unavailable, expect IncidentReport row with incident_type = 'NoEquipment'
Expected Result: Emergency patient prioritized, resource request created, incident logged if equipment unavailable.

Scenario C: Ward Admission with Available Bed

Steps:

Check bed availability: Navigate to dashboard, verify bed count for "Female Ward" > 0 (or seed more beds)
Check in patient:
Select unit: "Ward Admission - Female"
Click "Confirm Check-In"
Verify response: admission_reserved: true, bed_label: "FEMALE-01" (or similar)
Check database:
SELECT * FROM BedReservation WHERE visit_id = <visit_id> - expect row with status = 'Reserved'
SELECT * FROM Bed WHERE bed_id = <bed_id> - expect is_occupied = 1
Verify dashboard shows reduced bed count for Female Ward
Expected Result: Bed reserved atomically, availability decremented.

Scenario D: Ward Admission with NO Available Beds (Waitlist)

Steps:

Occupy all Female Ward beds (either manually via SQL: UPDATE Bed SET is_occupied = 1 WHERE unit_id = <female_unit_id>, or check in multiple patients)
Check in new patient:
Select unit: "Ward Admission - Female"
Notice banner: "No available beds in Female Ward"
Modal shows "Add to Waitlist?" option
Click "Add to Waitlist"
Verify response: admission_reserved: false, waitlist_id: <id>
Check database: SELECT * FROM AdmissionWaitlist WHERE visit_id = <visit_id> - expect row with status = 'Waiting', priority_level = 'High/Medium/Low'
Navigate to "Waitlist & Admissions" tab
Verify patient appears in waitlist table
Release a bed (manually or via another patient discharge)
Click "Notify Patient" button on waitlist entry
Verify waitlist status updates to "Notified"
Expected Result: Waitlist created when no beds available, notification workflow functional.

Edge Cases to Test
 Check-in without unit selection (should default to Outpatient)
 Resource request when resource already at zero (should create fulfilled = 0 request and incident)
 Concurrent bed reservation (2 users try to reserve last bed simultaneously) - should use transaction to prevent double-booking
 Waitlist notification when patient already notified (should be idempotent)
 Staff shift data missing (wait time estimation should fall back to default)
 View attendance records filtered by unit (verify new filter works)
Post-Implementation Recommendations
Operational Procedures
Daily Bed Audit: Receptionist runs morning check to sync physical bed availability with system (Bed.is_occupied field)
Shift Roster Entry: Administrator enters StaffShift records at start of each shift (or via future rostering UI)
Resource Threshold Configuration: Set critical thresholds (e.g., alert when oxygen < 2 units) in Resource table metadata
Weekly Incident Review: Clinical manager reviews IncidentReport table to identify procurement/staffing needs
Future Enhancements (Not in Scope)
WebSocket real-time notifications for emergency check-ins
Rostering UI for staff shift management
Bed discharge workflow (currently manual via BedReservation.status update)
Resource procurement requests triggered from incident reports
Historical analytics dashboard (trends over time)
Unit-specific queue boards for distributed displays
